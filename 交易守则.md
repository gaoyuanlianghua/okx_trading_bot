# OKX 交易守则

## 1. REST API 使用守则

### 1.1 请求验证
- **请求头必须包含**：
  - OK-ACCESS-KEY：API Key
  - OK-ACCESS-SIGN：使用 HMAC SHA256 哈希函数获得的哈希值，再使用 Base-64 编码
  - OK-ACCESS-TIMESTAMP：发起请求的时间（UTC），如：2020-12-08T09:08:57.715Z
  - OK-ACCESS-PASSPHRASE：创建 API 密钥时指定的 Passphrase
- **请求格式**：所有请求都应该含有 application/json 类型内容，并且是有效的 JSON

### 1.2 签名生成
- **签名算法**：对 timestamp + method + requestPath + body 字符串（+ 表示字符串连接），以及 SecretKey，使用 HMAC SHA256 方法加密，通过 Base-64 编码输出
- **示例**：sign=CryptoJS.enc.Base64.stringify(CryptoJS.HmacSHA256(timestamp + 'GET' + '/api/v5/account/balance?ccy=BTC', SecretKey))
- **说明**：
  - timestamp：与 OK-ACCESS-TIMESTAMP 请求头相同，为 ISO 格式
  - method：请求方法，字母全部大写：GET/POST
  - requestPath：请求接口路径，如：/api/v5/account/balance
  - body：请求主体的字符串，如果请求没有主体（通常为 GET 请求）则 body 可省略
  - SecretKey：用户申请 API Key 时所生成

### 1.3 GET 请求
- **用途**：获取账户信息、市场数据、订单信息等
- **示例**：
  - 获取账户余额：GET /api/v5/account/balance
  - 获取交易产品基础信息：GET /api/v5/account/instruments?instType=SPOT
  - 获取持仓信息：GET /api/v5/account/positions?instId=BTC-USDT
- **限速规则**：
  - 获取交易产品基础信息：20次/2s
  - 获取账户余额：10次/2s
  - 获取持仓信息：10次/2s

### 1.4 POST 请求
- **用途**：下单、修改订单、取消订单、设置杠杆等
- **示例**：
  - 下单：POST /api/v5/trade/order
  - 设置杠杆：POST /api/v5/account/set-leverage
- **请求体格式**：
  ```json
  {
    "instId": "BTC-USDT",
    "tdMode": "cash",
    "side": "buy",
    "ordType": "limit",
    "px": "1000",
    "sz": "0.01"
  }
  ```
- **限速规则**：
  - 子账户维度，每2秒最多允许1000个订单相关请求
  - 批量订单接口和单订单接口的限速是独立的

## 2. WebSocket (WS) 使用守则

### 2.1 连接
- **连接地址**：
  - 实盘 WebSocket 公共频道：wss://ws.okx.com:8443/ws/v5/public
  - 实盘 WebSocket 私有频道：wss://ws.okx.com:8443/ws/v5/private
  - 模拟盘 WebSocket 公共频道：wss://wspap.okx.com:8443/ws/v5/public
  - 模拟盘 WebSocket 私有频道：wss://wspap.okx.com:8443/ws/v5/private
- **连接限制**：3 次/秒 (基于 IP)
- **请求限制**：每个连接对于订阅/取消订阅/登录请求的总次数限制为 480 次/小时
- **保持连接**：
  1. 每次接收到消息后，设置一个定时器，定时 N 秒，N 小于 30
  2. 如果定时器被触发（N 秒内没有收到新消息），发送字符串 'ping'
  3. 期待一个文字字符串 'pong' 作为回应。如果在 N 秒内未收到，请发出错误或重新连接

### 2.2 登录
- **请求示例**：
  ```json
  {
    "op": "login",
    "args": [
      {
        "apiKey": "******",
        "passphrase": "******",
        "timestamp": "1538054050",
        "sign": "7L+zFQ+CEgGu5rzCj4+BdV2/uUHGqddA9pI6ztsRRPs="
      }
    ]
  }
  ```
- **签名生成**：先将 timestamp 、 method 、 requestPath 进行字符串拼接，再使用 HMAC SHA256 方法将拼接后的字符串和 SecretKey 加密，然后进行 Base64 编码
  - timestamp：Unix Epoch 时间戳，单位为秒
  - method：总是 'GET'
  - requestPath：总是 '/users/self/verify'

### 2.3 订阅
- **请求示例**：
  ```json
  {
    "op": "subscribe",
    "args": [
      {
        "channel": "tickers",
        "instId": "BTC-USDT"
      }
    ]
  }
  ```
- **频道类型**：
  - 公共频道：无需登录，包括行情频道，K线频道，交易数据频道，资金费率频道，限价范围频道，深度数据频道，标记价格频道等
  - 私有频道：需登录，包括用户账户频道，用户交易频道，用户持仓频道等

### 2.4 取消订阅
- **请求示例**：
  ```json
  {
    "op": "unsubscribe",
    "args": [
      {
        "channel": "tickers",
        "instId": "BTC-USDT"
      }
    ]
  }
  ```

### 2.5 下单
- **请求示例**：
  ```json
  {
    "id": "1512",
    "op": "order",
    "expTime": "1597026383085",
    "args": [
      {
        "side": "buy",
        "instId": "BTC-USDT",
        "tdMode": "isolated",
        "ordType": "market",
        "sz": "100"
      }
    ]
  }
  ```

## 3. 交易规则

### 3.1 订单类型
- 限价委托 (Limit)
- 市价委托 (Market)
- 只挂单 (Post only)
- 全部成交或立即取消 (FOK)
- 立即成交并取消剩余 (IOC)
- 市价委托立即成交并取消剩余 (optimal limit IOC)
- 止盈止损 (TP/SL)

### 3.2 订单限制
- 未成交订单（包括 post only，limit 和处理中的 taker 单）的最大挂单数：4,000 个
- 单个交易产品未成交订单的最大挂单数为 500 个
- 策略委托订单最大挂单数：
  - 止盈止损：100 个 每个 Instrument ID
  - 计划委托：500 个
  - 移动止盈止损：50 个
  - 冰山委托：100 个
  - 时间加权委托：20 个
- 网格策略最大个数：
  - 现货网格：100 个
  - 合约网格：100 个

### 3.3 交易限制
- 当 taker 订单匹配的 maker 订单数量超过最大限制 1000 笔时，taker 订单将被取消
- 限价单仅成交与 1000 笔 maker 订单相对应的部分，并取消剩余
- 全部成交或立即取消（FOK）订单将直接被取消

### 3.4 交易时效性
- 可以设置请求有效截止时间 expTime，格式为 Unix 时间戳的毫秒数
- 如果服务器当前系统时间超过 expTime，则该请求不会被服务器处理
- 支持接口：下单、批量下单、修改订单、批量修改订单

## 4. 限速规则

### 4.1 REST API 限速
- 公共未经身份验证的 REST 限速基于 IP 地址
- 私有 REST 限速基于 User ID（子帐户具有单独的 User ID）
- 交易相关 API（下订单、取消订单和修改订单）的限速在 REST 和 WebSocket 通道之间共享
- 下单、修改订单、取消订单的限速相互独立
- 限速在 Instrument ID 级别定义（期权除外）
- 期权的限速是根据 Instrument Family 级别定义的

### 4.2 WebSocket 限速
- WebSocket 登录和订阅限速基于连接
- WebSocket 订单管理限速基于 User ID（子账户具有单独的 User ID）

### 4.3 子账户限速
- 子账户维度，每 2 秒最多允许 1000 个订单相关请求
- 仅有新订单及修改订单请求会被计入此限制
- 对于包含多个订单的批量请求，每个订单将被单独计数
- 如果请求频率超过限制，系统会返回 50061 错误码

### 4.4 基于成交比率的子账户限速
- 适用于用户等级 >= VIP5 的用户
- 交易所将在每天 00:00 UTC，根据过去七天的交易数据计算成交比率
- 每日 08:00 UTC，系统将根据 UTC 时间 00:00 的数据快照，选取子账户成交比率及母账户合计成交比率中的较大值决定子账户的未来限速
- 成交比率越高，子账户的限速越高，最高可达 10,000 订单请求/2 秒

## 5. 最佳实践

### 5.1 限速管理
- 如果需要的请求速率高于限速，可以设置不同的子账户来批量请求限速
- 建议使用此方法来限制或间隔请求，以最大化每个帐户的限速并避免断开连接或拒绝请求

### 5.2 连接管理
- 保持 WebSocket 连接活跃，定期发送 ping 消息
- 处理连接断开的情况，实现自动重连机制

### 5.3 错误处理
- 处理 API 返回的错误码，特别是限速相关的错误码（50011、50061）
- 实现重试机制，对于临时错误进行有限次数的重试

### 5.4 安全措施
- 保护 API 密钥和 SecretKey，不要在代码中硬编码
- 使用环境变量或配置文件存储敏感信息
- 定期轮换 API 密钥

## 6. 常见错误码

| 错误码 | 描述 |
|--------|------|
| 50011 | 用户请求频率过快，超过该接口允许的限额 |
| 50061 | 子账户请求频率过快，超过子账户限速 |
| 60009 | Login failed |
| 64008 | The connection will soon be closed for a service upgrade. Please reconnect. |

## 7. 注意事项

- **模拟盘交易**：请求的 header 里面需要添加 "x-simulated-trading: 1"
- **返回数据规则**：
  - 当返回数据中，有 code，且没有 sCode 字段时，code 和 msg 代表请求结果或者报错原因
  - 当返回中有 sCode 字段时，代表请求结果或者报错原因的是 sCode 和 sMsg，而不是 code 和 msg
- **交易产品**：不同的交易产品有不同的交易规则和限速，需要根据实际情况进行调整
- **账户模式**：不同的账户模式（现货模式、合约模式、跨币种保证金模式、组合保证金模式）有不同的 API 接口和返回字段

## 8. 总结

- 遵循 OKX API 的请求验证和签名生成规则
- 合理管理 API 请求速率，避免触发限速
- 正确处理 WebSocket 连接和消息
- 实现完善的错误处理和重试机制
- 保护 API 密钥和 SecretKey 的安全

通过遵守以上交易守则，可以确保交易机器人能够稳定、高效地与 OKX API 进行交互，提高交易的成功率和可靠性。
